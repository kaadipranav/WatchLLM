GROUP B — WORKER ORCHESTRATION
================================================================================

Files to edit/create: watchllm/worker/chaos_worker.ts + new orchestrator_worker.ts

[x] B1 [P1] Parent Orchestrator Worker
        - Triggered by POST /api/simulate after simulation row is created
        - Reads agent config from Supabase (agents table)
        - Fans out one Attack Worker invocation per category from config.categories
        - Waits for all 6 to complete (Promise.all or Durable Objects)
        - Aggregates results: compute overall severity_score (weighted average or max)
        - Writes R2 reports/{simulation_id}/summary.json
        - Writes R2 reports/{simulation_id}/replay_manifest.json (ordered failure list)
        - Updates simulations row: status='completed', severity_score, completed_at,
          total_runs, failed_runs
        - On any worker error → update simulations.status = 'failed'

[x] B2 [P1] Attack Worker — update simulations row per run
        - Currently the worker inserts sim_runs rows but NEVER touches simulations
        - After each run: increment simulations.total_runs (and failed_runs if failed)
        - This is what feeds the dashboard progress bar
        - Use Supabase RPC or PATCH with increment logic to avoid race conditions

[x] B3 [P1] R2 reports/{simulation_id}/summary.json schema
        - Per context §7: aggregated scores per category
        - Shape: { simulation_id, categories: { prompt_injection: { runs, failures, avg_severity }, ... } }

[x] B4 [P1] R2 reports/{simulation_id}/replay_manifest.json schema
        - Per context §7: ordered list of failures for dashboard replay
        - Shape: { simulation_id, failures: [{ run_id, category, severity, r2_trace_key }] }

[x] B5 [P2] 30-second timeout on target agent fetch
        - callTargetAgent() has no AbortSignal/timeout
        - Add: const controller = new AbortController(); setTimeout(() => controller.abort(), 30_000)
        - Pass signal to fetch()
        - On abort: mark run as failed with explanation "Target agent timed out"

[x] B7 [P2] Category batching per Attack Worker
        - Current design: 6 workers × 1 run each → high startup overhead at scale
        - Improved design: 6 workers × (num_runs / 6) runs each
          e.g. 1000 runs → each worker handles ~166 runs sequentially before exiting
        - Workers already loop internally; change the outer dispatch to pass
          a run_count per worker instead of spinning up a new worker per run
        - Benefits: less Worker invocation overhead, better cost efficiency,
          more turns per seed prompt context
        - Orchestrator change: pass { category, run_count, start_index } per worker

[x] B6 [P2] Explicit error logging in Worker
        - On Groq failure, Anthropic failure, R2 write failure, Supabase write failure:
          log structured error with simulation_id + run_id + category
        - Workers: use console.error() — Cloudflare surfaces these in Workers Tail
